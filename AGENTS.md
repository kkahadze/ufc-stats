# AGENTS.md

This repo is a static UFC fight-network site plus a Python scraper that regenerates the data files the site reads.

Use this file as the local operating manual before making changes.

## What This Repo Is

- `docs/` is the shipped site. GitHub Pages serves this directory directly.
- `scripts/update_fight_data.py` is the only data-generation pipeline.
- `docs/data/*.json` are generated artifacts consumed by the frontend.
- `.cache/ufc-stats/` is a local scraper cache. It should speed up reruns but should not be committed.

There is no frontend build step, bundler, or framework. The site is plain HTML/CSS/JS with D3.

## Key Files

- `docs/index.html`
  The page shell, controls, legend, and script includes.
- `docs/main.js`
  Graph rendering, D3 force layout, filters, search spotlighting, tooltips, and zoom behavior.
- `docs/style.css`
  Entire visual system for the site.
- `docs/data/*.json`
  Division datasets generated by the scraper.
- `docs/flags/*.png`
  Local flag assets used by fighter nodes.
- `scripts/update_fight_data.py`
  Scrapes UFC Stats and UFC.com, resolves rankings and fighter flags, and writes `docs/data/`.
- `.github/workflows/update-fight-data.yml`
  Weekly Monday refresh job plus manual dispatch.
- `README.md`
  User-facing setup and data-source notes.

## Safe Assumptions

- The frontend reads JSON directly from `docs/data/`; if the JSON shape changes, `docs/main.js` usually needs to change with it.
- The scraper output is the source of truth for fighter metadata like rank, champion state, former champion state, bout details, and country/flag info.
- Most frontend work does not require touching the scraper.
- Most scraper work does require regenerating `docs/data/*.json`.

## Common Tasks

### Run the site locally

From repo root:

```bash
python3 -m http.server 4192 -d docs
```

Then open `http://127.0.0.1:4192`.

### Refresh fight data

```bash
python3 -m pip install -r requirements.txt
python3 scripts/update_fight_data.py
```

Force a full refresh and ignore cached pages:

```bash
python3 scripts/update_fight_data.py --refresh-all
```

### Validate frontend changes

```bash
node --check docs/main.js
```

If you changed HTML/CSS/JS behavior, also run the site locally and manually verify the interaction in a browser.

### Validate scraper changes

```bash
python3 -m py_compile scripts/update_fight_data.py
python3 scripts/update_fight_data.py
```

Check that the expected `docs/data/*.json` files changed and that no cache directory content was staged by mistake.

## Repo-Specific Rules

### Frontend

- Keep the site static. Do not introduce a framework or build tool unless explicitly requested.
- Preserve the current D3 architecture unless there is a strong reason to restructure it.
- The graph already supports:
  - division selection
  - roster filters
  - minimum fight-count filters
  - fighter spotlight search
  - node and edge tooltips
  - command-scroll zoom
- New interaction features should fit into the existing `docs/main.js` structure instead of layering in unrelated state systems.

### Scraper

- Prefer official UFC Stats and UFC.com sources already used by `scripts/update_fight_data.py`.
- Respect the cache structure. Do not remove incremental caching unless the user explicitly asks for it.
- If you add metadata to nodes or links, keep field names stable and update the frontend in the same change.
- If you add a manual fighter metadata override, put it in the existing override structures rather than scattering one-off logic through parsing code.

### Generated Data

- `docs/data/*.json` are generated files. If scraper logic changes, regenerate them in the same task.
- Do not hand-edit generated JSON unless the user explicitly asks for a one-off patch and understands it will be overwritten next refresh.

### Cache

- `.cache/ufc-stats/` is expected during local scraper runs.
- Never commit `.cache/`.
- If scraper behavior looks wrong, consider whether stale cache data is the cause before changing parsing logic.

## Workflow Notes

- The GitHub Action runs every Monday at `09:00 UTC` and can also be triggered manually.
- It restores `.cache/ufc-stats`, runs `python scripts/update_fight_data.py`, and commits `docs/data` only if those generated files changed.
- Avoid adding unstable fields to generated JSON that would create meaningless weekly diffs.

## Known Project Conventions

- Flag nodes use local PNG assets in `docs/flags/`, not remote flag URLs.
- Fighter identity should stay based on stable IDs where possible, not only display names.
- Current rankings and champion state come from the UFC rankings snapshot.
- Former champion and title-challenger state are division-specific metadata derived from title-bout history.
- Some fighter country cases are intentionally handled through manual overrides when UFC source data is incomplete or misleading.

## When Editing

- Keep comments sparse and only add them where the logic is not obvious.
- Prefer small, surgical changes over broad rewrites.
- If you touch `docs/index.html`, make sure IDs used by `docs/main.js` still exist.
- If you touch search, zoom, or tooltip behavior in `docs/main.js`, verify it does not break drag behavior.
- If you touch graph styling in `docs/style.css`, verify hover states, spotlight dimming, and status rings still read clearly on a light background.

## Pre-Completion Checklist

Before finishing a task, check the ones that apply:

- `node --check docs/main.js`
- `python3 -m py_compile scripts/update_fight_data.py`
- `python3 scripts/update_fight_data.py`
- local HTTP server smoke test
- generated `docs/data/*.json` reviewed
- `.cache/` not staged
- `README.md` updated if the workflow or developer-facing behavior changed

## Avoid

- Adding a frontend toolchain for small UI changes.
- Replacing official data sources with weaker third-party sources without a clear reason.
- Hardcoding one-off UI state in multiple places when the same behavior can live in `docs/main.js`.
- Hand-editing generated datasets as a substitute for fixing the scraper.
- Committing cache artifacts.
